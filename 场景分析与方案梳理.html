UX Portfolio Case｜t1 场景问题分析
01
1. 核心问题

用户看到很多线路，但仍然选不出来

表面问题

用户能看到距离、耗时、爬升、难度、评分和线路照片，但这些信息更多是在介绍线路

真正问题

用户需要判断的是：结合我的体力、目的、时间、交通、同行人员和当天环境，这条线路到底适不适合我

设计目标

把产品从“线路展示工具”，升级为“线路决策工具”：在每个关键环节帮助用户做判断，而不是让用户自己拼信息

02
2. 核心用户群体

先定义用户，再分析场景

2.1 新手独行

怕走不完，也怕不安全

平时很少徒步，想一个人出去放松，但担心迷路、信号、体力和突发风险

2.2 临时休闲

想快速出发

没有提前计划，只想找一条附近、轻松、交通方便、能当天完成的线路

2.3 家庭同行

要照顾最弱成员

带孩子、老人或婴儿车，不能只判断成年人能不能走

2.4 目的体验

不是为了走路

为了拍照、约会、看日落、赏花或城市观光，更关心沿途体验

2.5 进阶挑战

想找独特路线

能看懂基础数据，更关心挑战性、独特性、人流量和是否值得专门去

跨用户条件

限制会叠加

无车、临时出发、雨后路况、同行人员，都会改变一条线路是否合适

04
4.1 产生出行想法

临时想出去走走，却不知道该找什么线路

人物 / 场景

周日下午，用户心情不好，想立刻出去透透气。他没有装备，只想找一条两三个小时能结束、交通方便、有树荫和休息点的线路

需求 / 问题

他只有“轻松走走”的模糊想法，但不知道要筛距离、难度、交通还是耗时。热门线路越多，他越不知道从哪条开始看

建议改动方案

增加“现在出发 / 两小时轻松走 / 公交可达 / 无装备可走”等快捷入口，再用三步条件补充完成快速选线

  • 方案一：快捷场景入口
  • 方案二：三步快速选线
  • 方案三：AI 描述需求
最优方案

快捷场景入口＋三步条件补充，AI 作为复杂需求补充

现在出发→选时间→选交通→推荐可完成线路
05
4.2 浏览 / 搜索线路

线路很多，但每条看起来都差不多

人物 / 场景

新手搜索“北京周边徒步”，看到几十条线路。大部分都是 8-12 公里、两星难度、3-5 小时、评分 4.7 以上

需求 / 问题

用户看不出哪条适合一个人、哪条交通方便、哪条需要装备、哪条适合拍照。结果只能一条条点进去看

建议改动方案

重构线路卡片，让卡片直接展示“适合谁、为什么适合、有什么限制”，而不是只展示统一数据

  • 场景标签：新手友好、公交通达、亲子适合
  • 一句话判断：适合第一次独行，但后半段信号弱
  • 动态卡片：根据用户条件调整展示内容
最优方案

动态线路卡片＋一句话结论＋2-3 个关键标签

选择条件→重新排序→展示原因和限制→快速排除
06
4.3 初步筛选线路

看到了距离和爬升，却不知道自己能不能走完

人物 / 场景

久坐用户看到一条线路：10 公里、爬升 500 米、两星难度、预计 4 小时。他不知道 500 米爬升到底有多累

需求 / 问题

他卡在“保留还是排除”这一步。太谨慎会错过能走的线路，判断错误又可能到了现场才发现难度超出能力

建议改动方案

把专业数据翻译成用户能理解的体感说明，并结合经验等级给出适配判断

  • 通俗难度：前40分钟连续爬坡，普通体力会明显喘
  • 经验等级：第一次 / 偶尔 / 经常徒步
  • 个人档案：后期结合历史运动数据
最优方案

通俗说明＋经验等级适配，后期升级为个人历史判断

看数据→看体感说明→选经验→看最难路段
07
4.3 初步筛选线路

线路写着“简单”，但家长还是不敢带孩子去

人物 / 场景

一家三口周末出行，孩子还需要推婴儿车。家长看到“7公里、难度简单”，但仍然担心台阶、厕所、休息点、围栏和中途退出

需求 / 问题

成年人觉得简单，不代表儿童或婴儿车也适合。一旦判断错，可能走到一半孩子走不动，或者婴儿车无法通过

建议改动方案

增加同行人员设置，按同行中最弱成员重新判断线路，并生成家庭版路线

  • 亲子标签：婴儿车可通行、厕所较多
  • 同行设置：孩子年龄、是否携带婴儿车
  • 家庭版路线：缩短距离，优先经过休息点和安全路段
最优方案

同行人员设置＋家庭版路线＋设施地图

选择带孩子→补充情况→重算适配→确认家庭版路线
08
4.4 查看线路详情

一个人去徒步，却不知道这条线路是否安全

人物 / 场景

新手准备独自去北京周边徒步。线路距离和难度都合适，但她担心人少路段、信号死角、危险路段、蛇虫、求助和退出点

需求 / 问题

她卡在安全确认阶段：线路可能走得完，但不知道一个人走是否安全。不确定风险，就很可能放弃这条线路

建议改动方案

增加安全地图和独行模式，既帮助选择前判断风险，也覆盖实际徒步中的安全提醒

  • 安全标签：全程信号良好、夜间不建议独行
  • 安全地图：信号弱区、救援点、退出位置
  • 独行模式：行程分享、紧急联系人、偏航提醒
最优方案

安全地图＋独行模式

选择独行→看安全概况→设置联系人→出发监测
09
4.4 查看线路详情

想拍照、看日落，却不知道这条线路能不能满足

人物 / 场景

情侣想通过徒步约会，希望路线不要太累，沿途能赏花、拍照，最好能看到日落。但他们不知道打卡点在哪、几点到合适、能否缩短路线

需求 / 问题

用户卡在体验确认：知道有风景，但不知道能不能获得自己想要的约会和拍照体验

建议改动方案

增加体验地图和时间节奏规划，把“哪里值得停留”和“什么时候去”结合起来

  • 体验标签：日落观景、赏花、轻松约会
  • 体验地图：拍照点、休息区、咖啡馆
  • 时间规划：倒推建议出发时间和停留时长
最优方案

体验地图＋时间节奏规划

选择目的→看体验节点→算出发时间→生成缩短路线
10
4.4 查看线路详情

老手能看懂数据，却不知道这条线路值不值得去

人物 / 场景

有经验的用户想找人少、有挑战、地貌特别的线路。他能看懂海拔、距离和坡度，但不知道和自己走过的路线相比，是否真的有新鲜感

需求 / 问题

老手卡在价值判断：知道自己能走，但不知道是否值得专门去。如果所有线路只展示统一数据，就很难发现真正适合自己的路线

建议改动方案

增加历史线路对标和进阶探索模式，让老手判断线路差异，而不是自己消化更多专业数据

  • 进阶数据：坡度分布、技术路段、人流量
  • 历史对比：比你走过的XX难度更高
  • 探索模式：人少、高难度、特殊地貌、非跟团
最优方案

历史线路对标＋进阶探索模式

进阶模式→建立基准→推荐挑战路线→展示差异
11
4.5 对比候选线路

收藏了几条线路，反而越看越难选

人物 / 场景

用户收藏了三条距离和评分都很接近的线路。他想比较哪条更安全、交通更方便、适合拍照、人更少、适合孩子，但这些信息散落在不同详情页

需求 / 问题

用户卡在最终决策：信息很多，但无法直接比较。来回切换页面会增加记忆成本，最后可能随便选一条或继续拖延

建议改动方案

增加动态对比功能，根据用户目的自动选择比较维度，并给出推荐理由

  • 固定对比表：距离、难度、交通、补给
  • 自选维度：安全、亲子、拍照、人流
  • 系统结论：按当前目标推荐A或B
最优方案

动态对比＋推荐理由＋用户可修改条件

加入对比→选关键维度→看优势限制→获得推荐
12
4.6 出发前确认

线路本身合适，但现在出发可能来不及

人物 / 场景

用户下午两点临时想出去，看到线路预计 3 小时，觉得晚上能回来。但实际还要考虑到起点的时间、天黑、景区关闭、末班车和返程难度

需求 / 问题

用户卡在执行判断：线路平时能走，不代表现在出发还能走。一旦判断错误，可能到了起点才发现时间不够

建议改动方案

根据当前位置、当前时间、去返程和线路耗时，判断这条线路今天是否还能完成

  • 显示最晚出发时间
  • 计算完整往返时间
  • 根据当前时间推荐可完成线路
最优方案

实时完整行程判断

读取位置时间→算去程→算徒步返程→提示风险
13
4.7 实际徒步

知道怎么走，却不知道接下来最需要注意什么

人物 / 场景

用户已经开始徒步，可能会遇到信号死角、坡度突然变陡、距离下一个厕所很远、最佳拍照点即将错过，或按当前速度无法在天黑前完成

需求 / 问题

导航只解决方向问题，但没有持续帮助用户处理风险和体验重点。用户知道往哪走，却不知道接下来应该注意什么

建议改动方案

增加关键节点提醒和动态行程调整，根据位置、速度、时间和环境持续更新提醒

  • 地图静态标记
  • 关键节点前提醒
  • 根据速度和时间动态调整行程
最优方案

关键节点提醒＋动态行程调整

开始线路→展示下一段重点→节点提醒→动态调整

6. 方案优先级

先解决“选不出来”，再解决“选了去不了”

01

第一优先级：解决选不出来

场景化选线、动态线路卡片、通俗难度解释、个人适配判断、线路对比

02

第二优先级：解决选了但去不了

时间判断、公共交通往返、同行人员适配、天气和实时路况

03

第三优先级：提升实际体验

独行安全模式、体验地图、关键节点提醒、动态行程调整

16
7. 最终设计方向

不要只做“AI 推荐线路”，而是解决完整决策问题

先理解用户这次为什么出发，再结合经验、时间、交通和同行人员筛选线路；选中后确认当天是否能顺利执行；实际徒步时再帮助用户避开风险、获得想要的体验

选前

帮用户明确目的、筛选线路、理解是否适合自己

出发前

帮用户确认交通、时间、天气、装备和同行风险

途中

帮用户识别关键节点、避开风险、动态调整行程
